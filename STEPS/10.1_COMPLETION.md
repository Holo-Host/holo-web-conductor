## Overview

Step 10.1 focuses on creating an AppClient adapter (FishyAppClient) that allows existing Holochain web apps using @holochain/client to work with the Fishy browser extension without code changes.

## What Was Accomplished

### 1. FishyAppClient Implementation

**Location**: `/home/eric/code/metacurrency/holochain/ziptest/ui/src/fishy/FishyAppClient.ts`

Created a drop-in replacement for @holochain/client's AppClient that uses the window.holochain API provided by Fishy extension.

**Key Features**:
- Implements full `AppClient` interface from @holochain/client
- Handles connection and hApp installation via Fishy extension
- Translates callZome requests to extension API calls
- Deep conversion of Chrome messaging arrays back to Uint8Array
- `cachedAppInfo` property for ZomeClient signal filtering compatibility

**Interface Methods Implemented**:
- `callZome()` - Call zome functions via extension
- `appInfo()` - Returns standard AppInfo format
- `on("signal", callback)` - Signal subscription
- `myPubKey` / `installedAppId` getters
- Stub implementations for clone cell methods (not supported in Fishy mode)

### 2. ZeroArcProfilesClient

## Known Issues / Remaining Work

### Critical: Remote Signal Architecture (Step 10.2)
The current remote signal implementation forwards signals directly to UI tabs instead of invoking the WASM's `recv_remote_signal` callback. This bypasses the Holochain architecture where the WASM decides whether to forward signals to the UI via `emit_signal()`.

**See**: `STEPS/10.2_PLAN.md` for the fix plan.

### TypeScript Compile Error
```
src/ribosome/index.ts(376,7): error TS2353: Object literal may only specify known
properties, and 'remoteSignals' does not exist in type 'ZomeCallResult'.
```
Need to add `remoteSignals` to `ZomeCallResult` interface.

### Signal Format
FishyAppClient needs to ensure signals are formatted correctly:
- `type: "app"` (not "remote_signal")
- `value.cell_id` (not dna_hash/from_agent separately)
- `value.zome_name`
- `value.payload`

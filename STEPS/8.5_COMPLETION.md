# Step 8.5 Completion: Integration & Publish Workflow

**Date Completed**: 2026-01-06
**Status**: COMPLETE

## Goal

Wire up automatic publishing of DhtOps after zome call commits, completing the end-to-end publish flow from browser extension to gateway.

## What Was Done

### 1. Ribosome Worker Transport (ribosome-worker.ts)

Updated the worker to include `pendingRecords` in its response when a zome call results in committed data:

```typescript
// Convert pending records for transport (Uint8Array -> Array)
let pendingRecordsForTransport: any[] | undefined;
if (zomeResult.pendingRecords && zomeResult.pendingRecords.length > 0) {
  pendingRecordsForTransport = zomeResult.pendingRecords.map(record => ({
    signed_action: {
      hashed: {
        content: record.signed_action.hashed.content,
        hash: Array.from(record.signed_action.hashed.hash),
      },
      signature: Array.from(record.signed_action.signature),
    },
    entry: record.entry ? {
      Present: record.entry.Present ? {
        entry_type: record.entry.Present.entry_type,
        entry: Array.from(record.entry.Present.entry),
      } : undefined,
    } : undefined,
  }));
}
```

### 2. Offscreen Document Integration (offscreen/index.ts)

Added PublishService integration to automatically publish pending records after each zome call:

- Added `PublishService` import from `@fishy/core/dht`
- Added `transportedRecordToRecord()` function to convert Chrome message format back to proper Record types
- Added `publishPendingRecords()` async function that:
  - Initializes PublishService on first use
  - Converts transported records back to Holochain Record format
  - Publishes each record via PublishService
- Updated `executeZomeCall()` to trigger publishing asynchronously after returning the result

### 3. Publish Flow Architecture

```
Zome Call (create entry)
    │
    ▼
Host Function (create.ts)
    │
    ├─► Store action/entry in IndexedDB
    │
    └─► Add to pendingRecords list
          │
          ▼
    Ribosome returns zomeResult with pendingRecords
          │
          ▼
    Worker converts pendingRecords for Chrome transport
          │
          ▼
    Offscreen receives result
          │
          ├─► Returns result to caller immediately
          │
          └─► Background: publishPendingRecords()
                    │
                    ▼
              PublishService.publishRecord()
                    │
                    ▼
              HTTP POST /dht/{dna}/publish
                    │
                    ▼
              Gateway receives, stores in TempOpStore
                    │
                    ▼
              Kitsune2 publish_ops() to DHT authorities
```

## Known Limitations

1. **Single-node testing**: Without multiple conductors in the network, ops won't actually reach DHT authorities. The publish flow is complete but final delivery requires a real DHT network.

2. **No retry UI**: PublishTracker stores retry state in IndexedDB but there's no UI to show publish status to users. This could be added in a future step.

3. **Asynchronous publishing**: Publishing happens in the background after the zome call returns. If publishing fails, the user won't immediately know. Failed publishes are tracked and can be retried.

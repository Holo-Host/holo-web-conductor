# Step 8.5: Integration & Publish Workflow

## Goal

Wire up automatic publishing after each zome call commit. The complete flow from host function commit to DHT publishing.

## Complete Workflow

```
1. Zome call executes (e.g., create_entry)
   ↓
2. Host function commits action + entry to local storage
   ↓
3. Generate DhtOps from committed record (Step 8.1)
   ↓
4. Sign each DhtOp with agent key (Step 8.2)
   ↓
5. Queue signed ops for publishing (Step 8.4)
   ↓
6. Send to gateway publish endpoint (Step 8.3)
   ↓
7. Update publish status on response
   ↓
8. Retry on failure with exponential backoff
```

## Integration Points

### Host Functions

After each commit, trigger publish workflow:

```typescript
// In create.ts, update.ts, delete.ts, create_link.ts, delete_link.ts
storage.putAction(action, dnaHash, agentPubKey);
storage.putEntry(entry, dnaHash, agentPubKey);

// Trigger publish (async, non-blocking)
publishService.queueForPublish(action, entry);
```

### Publish Service

Central service that coordinates publishing:

```typescript
class PublishService {
  async queueForPublish(action: Action, entry?: Entry): Promise<void>;
  async processQueue(): Promise<void>;
  async retryFailed(): Promise<void>;
}
```

4. Handle async publish (don't block zome call return)

## Dependencies

- Step 8.0 through 8.4 must be complete
- Gateway must be running with kitsune2 enabled
- Conductor must be connected to gateway

---

## Success Criteria

1. Zome call returns immediately (publish is async)
2. DhtOps are queued and sent to gateway
3. Gateway ACK updates publish status
4. Failed publishes are retried
5. Published data is visible to other Holochain nodes

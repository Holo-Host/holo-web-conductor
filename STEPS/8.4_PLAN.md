# Step 8.4: Publish Tracking in Extension

## Goal

Track publish status in the browser extension to ensure data reaches the DHT. Implement retry logic for failed publishes.

## Publish Status States

```typescript
enum PublishStatus {
  Pending = 'pending',     // Committed locally, not yet published
  Publishing = 'publishing', // Currently being sent to gateway
  Published = 'published',   // Gateway acknowledged receipt
  Failed = 'failed',         // Publish failed, needs retry
}
```

## Storage Schema

Add to IndexedDB:

```typescript
interface PendingPublish {
  opHash: Uint8Array;       // Hash of the DhtOp
  actionHash: Uint8Array;   // Source action
  opType: string;           // StoreRecord, StoreEntry, etc.
  status: PublishStatus;
  createdAt: number;        // Timestamp
  lastAttempt: number | null;
  attemptCount: number;
  error: string | null;
}
```

## Retry Strategy

- Exponential backoff: 1s, 2s, 4s, 8s, 16s, 32s, max 60s
- Max retry attempts: 10
- Retry on network errors, gateway unavailable
- Don't retry on signature/validation errors (permanent failure)

---

## Implementation Tasks

1. Add PublishStatus enum to storage types
2. Create pending_publishes table in IndexedDB
3. Track publish attempts and status
4. Implement retry logic with exponential backoff
5. Clean up successful publishes after confirmation

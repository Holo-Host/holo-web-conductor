## Architecture Notes

### Two-Phase Publish Process

1. **Publish notification**: `publish_ops(op_ids, target)` sends only op IDs to target peers
2. **Fetch cycle**: Target peers fetch actual op data back from gateway's TempOpStore

This means the gateway must hold ops in TempOpStore until DHT authorities fetch them.

### Zero-Arc Node Implications

- Gateway is zero-arc (`DhtArc::Empty`) - never an authority for ops
- Once DHT authorities store ops, they become the sources
- Gateway's TempOpStore can safely delete ops after fetch (or 60s timeout)

# Step 19.1 Issue: get_joining_timestamp_for_agent Deserialization Failure

## Status: Open ÔÇö Non-blocking for e2e test

## Symptom

The `profiles::get_joining_timestamp_for_agent` zome function receives WASM input bytes `[129, 162, 79, 107, 144]` which decode to `{"Ok": []}` (msgpack). The function expects `ZomeFnInput<AgentPubKey>` (an object with `input: <39-byte hash>` and `local: bool`). This causes a WASM deserialization error.

The error is cosmetic ÔÇö it prevents the "Joined" date from appearing on profile pages but does not block core mewsfeed functionality (creating mews, searching hashtags, following).

## What We Know

### 1. The error bytes are ExternIO inner bytes, not raw WASM bytes

The `#[hdk_extern]` macro (`map_extern_preamble!` in `holochain/crates/hdi/src/map_extern.rs`) does two-step deserialization:
1. `host_args::<ExternIO>(ptr, len)` ÔÇö deserializes raw WASM memory as ExternIO (serde_bytes handles bin8 format)
2. `extern_io.decode::<ZomeFnInput<AgentPubKey>>()` ÔÇö deserializes ExternIO inner bytes as the target type

The error occurs at step 2. The bytes `[129, 162, 79, 107, 144]` are the ExternIO **inner** bytes (after bin8 extraction), not the raw WASM memory. This confirms:
- `serializeToWasm`'s `encode(Uint8Array)` Ôćĺ bin8 wrapping is **correct** (ExternIO expects it)
- The background/ribosome pipeline correctly delivers bytes to WASM
- The problem is that the **content** of those bytes is wrong: `{Ok: []}` instead of `{input: <hash>, local: false}`

### 2. Confirmed: `encode({Ok: []})` = `[129, 162, 79, 107, 144]`

```
Node.js verification:
encode({Ok: []}) Ôćĺ Uint8Array [129, 162, 79, 107, 144]
  129 = fixmap(1)
  162 = fixstr(2) "Ok"
  144 = fixarray(0)
```

### 3. UI call sites had bugs (FIXED)

Two places call this function:
- `AgentProfileDetail.vue:55`: `payload: wrapInput(props.agentPubKey)` ÔÇö correct (Uint8Array)
- `AgentProfile.vue:235`: `payload: wrapInput(route.params.agentPubKey)` ÔÇö **BUG: passed base64 string instead of decoded Uint8Array**

**Fixed**: Changed `route.params.agentPubKey` to `agentPubKey.value` in AgentProfile.vue (3 occurrences: `get_joining_timestamp_for_agent`, `count_creators_for_follower`, `count_followers_for_creator`).

However, this string bug would NOT produce the `{Ok: []}` payload ÔÇö a base64 string would encode to a much larger msgpack payload. The `{Ok: []}` mystery remains.

`wrapInput(x)` returns `{input: x, local: false}` when `IS_FISHY` is true. If `agentPubKey` were a proper 39-byte Uint8Array, the encoded payload would be ~50+ bytes, not 5.

### 4. ProfilesClient is not involved

The `@holochain-open-dev/profiles` `ProfilesClient` class (v0.601.2) does NOT have a `get_joining_timestamp_for_agent` method. This is a custom mewsfeed function called directly via `client.callZome()`.

### 5. This is NOT a `serializeToWasm` or ExternIO bug

The double-encoding path (`encode()` in background Ôćĺ `encode()` in `serializeToWasm`) is correct by design. The WASM's `host_args::<ExternIO>()` uses serde_bytes to strip the bin8 wrapper, then `extern_io.decode()` decodes the inner bytes.

## Open Questions

1. **How does `{Ok: []}` become the payload?** The `wrapInput()` call should produce `{input: agentPubKey, local: false}`. Even if `agentPubKey` were `{Ok: []}`, wrapping would produce `{input: {Ok: []}, local: false}` (~20+ bytes), not 5.

2. **Is the agentPubKey prop correct?** Need to verify what `props.agentPubKey` actually contains at runtime. It may be receiving an unwrapped zome call result instead of a hash.

3. **Is this mewsfeed-specific or a general regression?** The ziptest profiles flow does NOT call `get_joining_timestamp_for_agent`. Need to check whether standard `get_agent_profile` (which also takes an AgentPubKey) works correctly.

4. **Could the offscreen Ok/Err unwrapping interfere?** The offscreen (lines 868-916) unwraps `{Ok: value}` from zome call results. If a result is being used as a subsequent call's payload, the unwrapping could strip the `{Ok: ...}` wrapper from the result but leave inner data in unexpected form.

## Likely Hypotheses (untested)

**Hypothesis A: The payload IS `{Ok: []}` from the page.** Something in the callZome pipeline strips or transforms the payload before it reaches the background. The 5-byte payload `{Ok: []}` suggests the background received `{Ok: []}` as `normalizedPayload` and encoded it directly.

**Hypothesis B: A previous zome call result is being passed as agentPubKey.** If a Vue component receives a result like `{Ok: <Uint8Array>}` from a zome call and passes it as a prop without unwrapping, and then Chrome messaging converts the Uint8Array to `[]` (empty array if objectified and not converted back), the payload could become `{Ok: []}`.

### 6. Full callZome pipeline traced (no transformation bugs found)

Traced the complete path: Page `wrapInput()` Ôćĺ FishyAppClient.callZome Ôćĺ inject `sendToContentScript` Ôćĺ content script Ôćĺ background `handleCallZome` Ôćĺ `normalizeUint8Arrays(payload)` Ôćĺ `encode(normalizedPayload)` Ôćĺ offscreen Ôćĺ worker Ôćĺ WASM.

No payload transformation or stripping was found. `normalizeUint8Arrays` correctly restores Uint8Arrays from Chrome's `{0: x, 1: y}` format. The pipeline should produce correct bytes for a valid Uint8Array input.

The `{Ok: []}` (5 bytes) origin remains unexplained. Since `wrapInput` wraps ALL payloads as `{input: x, local: false}`, even a bad input would produce `{input: <bad>, local: false}` (>5 bytes). Something must be bypassing `wrapInput` or the entire payload is being replaced with `{Ok: []}` at some point.

## Recommended Investigation

1. Add a diagnostic log in `background/index.ts` `handleCallZome` specifically for `fn_name === "get_joining_timestamp_for_agent"` to capture the raw `requestPayload.payload` BEFORE normalization
2. Re-test after the stringÔćĺUint8Array fix to see if the error still occurs
3. Check if Vue Query deduplication between AgentProfile.vue and AgentProfileDetail.vue could be involved (same query keys, different fetch functions)

## Impact

- **Not blocking**: The "Joined" date simply doesn't appear on profile pages
- **User-facing**: No error visible to user (Vue `useQuery` catches error silently via `watch(errorJoinedTimestamp, console.error)`)
- **E2E test**: Does not affect hashtag search test flow

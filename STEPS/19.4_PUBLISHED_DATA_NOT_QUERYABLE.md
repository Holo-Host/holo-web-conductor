# Step 19.4: Published DhtOps Not Returned by Queries

## Status: Open — Blocking mewsfeed e2e test

## Repo: hc-membrane (kitsune-dht-ops branch) + possibly holochain conductor

## Symptom

DhtOps are successfully published to conductors (12 ops to 2 peers, including CreateLink ops for hashtags), but subsequent `get_links` queries for the same data return 0 results.

### Evidence

```
# Publishing succeeds:
publish: Sent 12 ops to 2 peers (Create, CreateLink for #testmew hashtag)

# Querying returns nothing:
dht_get_links: 0 results for base=uhCEk... link_type=...
```

## Relationship to Step 19.3

This issue may be entirely explained by Step 19.3 (query-response timeout). If conductors never respond to queries, we'd see 0 results regardless of whether the data was actually stored.

**Resolve Step 19.3 first**, then re-test. If queries start returning responses but still show 0 results, this step becomes relevant.

## Possible Independent Causes

If Step 19.3 is resolved and queries still return 0:

### 1. Published data not integrated into DHT store

Conductors may accept published DhtOps via `send_notify` but not integrate them into their local DHT store. Check:
- Are `StoreRecord` and `RegisterAddLink` ops actually written to the conductor's database?
- Is there a validation step that rejects the ops silently?

### 2. Query routing mismatch

The `GetLinksReq` may query a different basis hash or link type than what was published. Verify:
- Base hash in query matches the base hash of the published CreateLink
- Link type filter matches the published link's type index
- Zome index is correct

### 3. Timing / integration delay

Published ops may need time to be integrated before they're queryable. Check if there's an async integration pipeline in the conductor that hasn't completed before the query arrives.

## Investigation Plan

1. Resolve Step 19.3 first
2. If queries return responses but 0 results:
   a. Query conductor directly via `curl` to its admin API to verify data exists
   b. Compare published base hashes with queried base hashes
   c. Check conductor logs for op integration/validation messages

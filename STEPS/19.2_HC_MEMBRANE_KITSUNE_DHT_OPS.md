# Step 19.2: Implement All DHT Operations via Kitsune in hc-membrane

## Status: Planned — Blocking mewsfeed e2e test

## Goal

Implement all DHT query operations in hc-membrane's kitsune path (`#[cfg(not(feature = "conductor-dht"))]`), removing the dependency on the `dht_util` conductor zome. Currently, `dht_get_details` has NO kitsune implementation — it always calls `call_dht_util("dht_get_details")`, which returns 502 for any hApp that doesn't ship a `dht_util` zome (including mewsfeed).

Additionally, the `count_links` and `get_agent_activity` wire protocol operations have no endpoints at all.

## Root Cause (from Step 19 Investigation)

hc-membrane's `/dht/{dna}/details/{hash}` endpoint lacks the `#[cfg(not(feature = "conductor-dht"))]` variant that `/dht/{dna}/record/{hash}` and `/dht/{dna}/links` have. It always routes through the conductor's `dht_util` zome, which:
1. Requires the hApp to include a `dht_util` coordinator zome
2. Returns `502: Zome 'dht_util' not found` for apps like mewsfeed

This causes mewsfeed's `get_mew_with_context_internal()` to silently fail: `get_details` returns 502 → cascade returns null → WASM error → `.filter_map(.ok())` drops all results → empty array.

## Holochain Wire Protocol Reference

From `holochain/crates/holochain_p2p/src/types/wire.rs`, the full set of DHT query messages:

```rust
enum WireMessage {
    // Already implemented in hc-membrane
    GetReq { msg_id, to_agent, dht_hash }
    GetRes { msg_id, response: WireOps }
    GetLinksReq { msg_id, to_agent, link_key, options }
    GetLinksRes { msg_id, response: WireLinkOps }
    RemoteSignalEvt { to_agent, zome_call_params_serialized, signature }
    ErrorRes { msg_id, error }

    // Need implementation
    CountLinksReq { msg_id, to_agent, query: WireLinkQuery }
    CountLinksRes { msg_id, response: CountLinksResponse }
    GetAgentActivityReq { msg_id, to_agent, agent, query, options }
    GetAgentActivityRes { msg_id, response: AgentActivityResponse }
    MustGetAgentActivityReq { msg_id, to_agent, agent, filter }
    MustGetAgentActivityRes { msg_id, response: MustGetAgentActivityResponse }

    // Not needed for gateway (peer-to-peer only)
    CallRemoteReq/Res
    SendValidationReceiptsReq/Res
    PublishCountersignEvt
    CountersigningSessionNegotiationEvt
}
```

## Implementation Plan

### Priority Order

1. **`get_details`** — Critical. Blocks mewsfeed e2e test. Uses existing `GetReq`/`GetRes` wire protocol (same as `get`).
2. **`count_links`** — High. Mewsfeed calls `count_links` per mew for like/reply/mewmew counts. New wire protocol + endpoint.

### Task 1: `get_details` via Kitsune (CRITICAL)

**Key insight**: `get_details` uses the SAME wire protocol as `get` (`GetReq`/`GetRes` with `WireOps`). The difference is only in how the response is formatted for the HTTP API. The existing `DhtQuery::get()` already does the right kitsune query — the endpoint just needs to format the `WireOps` response differently.

`WireOps` is an enum with two variants:
- `WireOps::Entry(WireEntryOps)` — contains `entry`, `creates`, `deletes`, `updates`
- `WireOps::Record(WireRecordOps)` — contains `action`, `entry`, `deletes`, `updates`

For `get_details`, we need to convert `WireOps` into the `Details` format that holochain returns:
- Entry hash → `Details::Entry { entry, actions, rejected_actions, deletes, updates, entry_dht_status }`
- Action hash → `Details::Record { record, validation_status, deletes, updates }`

**Files to change**:

1. **`src/routes/dht.rs`** — Add `#[cfg(not(feature = "conductor-dht"))]` variant of `dht_get_details`:
   ```rust
   #[cfg(not(feature = "conductor-dht"))]
   pub async fn dht_get_details(
       Path(path): Path<RecordPath>,
       State(state): State<AppState>,
   ) -> HcMembraneResult<Json<serde_json::Value>> {
       let dht_query = state.dht_query.as_ref()
           .ok_or_else(|| HcMembraneError::Internal("DHT queries not available".to_string()))?;
       let dna_hash = parse_dna_hash(&path.dna_hash)?;
       let hash = parse_any_dht_hash(&path.hash)?;
       let result = dht_query.get(&dna_hash, hash).await?;
       match result {
           Some(wire_ops) => {
               let json_value = wire_ops_to_details_json(&wire_ops);
               Ok(Json(json_value))
           }
           None => Ok(Json(serde_json::Value::Null)),
       }
   }

   #[cfg(feature = "conductor-dht")]
   pub async fn dht_get_details(
       // ... existing conductor implementation unchanged ...
   )
   ```

2. **`src/routes/dht.rs`** — Add `wire_ops_to_details_json()` conversion function:
   - For `WireOps::Entry(entry_ops)`: Build the Entry details structure with `entry`, `actions` (from `creates`), `rejected_actions: []`, `deletes`, `updates`, `entry_dht_status: "Live"`
   - For `WireOps::Record(record_ops)`: Build the Record details structure with `record` (from `action` + `entry`), `validation_status: "Valid"`, `deletes`, `updates`
   - Must produce JSON matching what Holochain's `get_details` returns — this is what fishy's `get_details.ts` host function expects

   Reference: Look at how fishy's `get_details.ts` (lines 122-163 for entry, lines 210-247 for record) constructs the Details structure from local storage. The hc-membrane endpoint must produce the same shape.

   Also reference: The existing `wire_ops_to_json()` function shows how to serialize `WireOps` — the details variant needs to restructure it into `{type: "Entry"|"Record", content: {...}}` format.

**No changes needed to `dht_query.rs`** — it already sends `GetReq` and receives `GetRes` with `WireOps`.

## Fishy-Side Changes (after hc-membrane is fixed)

### `count_links` host function needs cascade/network support

`fishy-step19/packages/core/src/ribosome/host-fn/count_links.ts` currently only queries local storage:
```typescript
let storedLinks = storage.getLinks(input.base, dnaHash, agentPubKey, linkTypeFilter);
```

Needs to use cascade like `get_links.ts` does:
```typescript
const cascade = new Cascade(storage, getNetworkCache(), getNetworkService());
const links = cascade.fetchLinks(dnaHash, agentPubKey, input.base, linkTypeFilter);
```

Also needs a `countLinksSync` implementation in `sync-xhr-service.ts` that calls the new `/dht/{dna}/count_links` endpoint, and the cascade should prefer the direct count endpoint over fetching all links and counting locally.

### Alternatively (simpler): Just count from get_links cascade

Since `get_links` already works via cascade, `count_links` could simply call the same cascade `fetchLinks` and count the results. This avoids needing a new endpoint entirely but is less efficient for large link sets.

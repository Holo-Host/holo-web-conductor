## Summary

Fixed the remote signal architecture to match Holochain's design. Both outgoing and incoming signals now follow the correct flow.

## What Was Fixed

### 1. Outgoing Signals (send_remote_signal)

**Before (wrong)**:
```
WASM send_remote_signal â†’ queue in callContext â†’ return in ZomeCallResult â†’ offscreen sends via WebSocket
```

**After (correct)**:
```
WASM send_remote_signal â†’ queue in callContext â†’ worker postMessage â†’ offscreen sends via WebSocket (fire-and-forget)
```

Key change: Signals are sent during the zome call as a side effect, not returned from `callZome`. This mirrors Holochain's `tokio::spawn` pattern.

### 2. Incoming Signals (recv_remote_signal)

**Before (wrong)**:
```
Gateway WebSocket â†’ Background handleRemoteSignal â†’ directly to UI tabs
```

**After (correct)**:
```
Gateway WebSocket â†’ Background handleRemoteSignal â†’ WASM recv_remote_signal â†’ emit_signal â†’ UI tabs
```

Key change: Remote signals are delivered by invoking the WASM's `recv_remote_signal` callback. The WASM decides whether to forward to UI by calling `emit_signal()`.

## Architecture Details

### Outgoing Signal Flow
1. WASM calls `send_remote_signal` host function
2. Host function signs signals and queues in `callContext.remoteSignals`
3. After zome call, `callZome` returns with `remoteSignals` array
4. Worker immediately sends `SEND_REMOTE_SIGNALS` message to offscreen
5. Offscreen's `handleSendRemoteSignals` forwards to WebSocket
6. WebSocket sends to gateway for delivery to target agents

### Incoming Signal Flow
1. Gateway receives signal from kitsune2 network
2. WebSocket forwards to offscreen document
3. Offscreen sends `REMOTE_SIGNAL` message to background
4. Background finds context for DNA hash
5. Background builds ZomeCallRequest for `recv_remote_signal`
6. Background executes via offscreen (WASM invocation)
7. WASM's `recv_remote_signal` may call `emit_signal()`
8. Any emitted signals are delivered to tabs matching the context's domain

## Additional Fix: call_info Host Function

The `call_info` host function was also rewritten as part of this step because `recv_remote_signal` invokes it to get caller information.

**Issues fixed**:
1. Missing `function_name` and `as_at` fields in return structure
2. `cap_grant` was `null` instead of proper `CapGrant::ChainAuthor` variant
3. Used wrong property names (`hash`/`seq` instead of `actionHash`/`actionSeq`) from `ChainHead`
4. BigInt timestamp wasn't being converted to number for msgpack serialization

**Correct `CallInfo` structure**:
```typescript
interface CallInfo {
  provenance: AgentPubKey;
  function_name: string;
  as_at: [ActionHash, number, number]; // hash, seq, timestamp_micros
  cap_grant: CapGrant;  // { ChainAuthor: AgentPubKey } or { RemoteAgent: ZomeCallCapGrant }
}
```

## Edge Cases Handled

2. **No tabs for domain**: Warning logged, signals not delivered
3. **recv_remote_signal not exported**: WASM error logged but not fatal
4. **WebSocket not connected**: Warning logged, outgoing signals dropped

## Gateway Optimizations

### Browser Agent Direct Delivery
When `send_remote_signals()` is called, the gateway now first checks if the target agent is a registered browser agent. If so, it delivers directly via WebSocket instead of routing through kitsune2. This optimization:
- Reduces latency (no kitsune2 peer discovery needed)
- Works even if kitsune2 connectivity is impaired
- Only falls back to kitsune2 for non-browser targets

### to_agent Field
The `ServerMessage::Signal` now includes a `to_agent` field identifying which local agent the signal is addressed to. This is needed when:
- Multiple agents from the same DNA are registered
- The extension needs to route signals to the correct WASM instance

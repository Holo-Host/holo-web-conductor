# Step 19.5: Reduce Sync XHR Default Timeout

## Status: Obsolete — `sync-xhr-service.ts` was removed; the ribosome worker uses its own inline ProxyNetworkService with direct XHR

## Repo: fishy-step19

## Symptom

When a gateway DHT query takes 30 seconds to timeout (Step 19.3), the sync XHR worker is blocked for the entire 30 seconds. Since the worker is single-threaded and processes requests serially, one slow query blocks ALL subsequent queries, causing a cascade of timeouts across the entire extension.

### Impact

- Alice's mew creation takes 30+ seconds to display because `get_links` queries after the create each block for 30s
- Bob's hashtag search queues up multiple queries that each wait 30s, hitting the test's 120s timeout
- callZome diagnostics in the e2e test all timeout because they're queued behind slow queries

## Proposed Change

### Reduce default timeout from 30s to 5-10s

**File**: [sync-xhr-service.ts](packages/core/src/network/sync-xhr-service.ts)

The default timeout is currently 30000ms (30 seconds). For zero-arc nodes that fetch all data from the network:
- A healthy gateway response should arrive in <2 seconds
- If no response in 5-10 seconds, the query is likely failing
- Waiting 30s just delays the cascade failure

```typescript
// Current:
const DEFAULT_TIMEOUT = 30000;

// Proposed:
const DEFAULT_TIMEOUT = 8000; // 8 seconds — enough for healthy responses, fast-fail on errors
```

**File**: [ribosome-worker.ts](packages/extension/src/offscreen/ribosome-worker.ts)

Mirror the same timeout change in the extension's inline network service copy.

### Consider per-endpoint timeouts

Some operations are inherently faster than others:
- `get_links` / `count_links`: Should be fast (<2s)
- `get` / `get_details`: May involve multi-hop lookups (<5s)
- `publish`: Fire-and-forget, no response timeout needed

Could add endpoint-specific timeouts, but the simpler approach of a single reduced default is probably sufficient for now.

### Do NOT reduce auth endpoint timeouts

The auth endpoints (`requestChallenge`, `verifyChallenge`) already have timeout DISABLED because they use a different flow (WebSocket-based challenge). These should remain unchanged.

## Key Files

| File | Purpose |
|------|---------|
| [sync-xhr-service.ts](packages/core/src/network/sync-xhr-service.ts) | Default timeout constant |
| [ribosome-worker.ts](packages/extension/src/offscreen/ribosome-worker.ts) | Mirror timeout in extension copy |

## Success Criteria

- Network failures surface within 8 seconds instead of 30 seconds
- Worker throughput improved: failed queries don't block the pipeline for 30s each
- Auth flow (requestChallenge/verifyChallenge) unaffected

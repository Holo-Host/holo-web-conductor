# Step 10.2: Fix Remote Signal Architecture

## Problem Statement

Remote signals are being forwarded **directly to the UI**, bypassing the WASM entirely. This is architecturally incorrect.

**Current (wrong) flow:**
```
Gateway WebSocket → Offscreen → Background handleRemoteSignal() → chrome.tabs.sendMessage → UI
```

**Correct flow (per Holochain):**
```
Gateway WebSocket → Offscreen → Background → invoke WASM recv_remote_signal()
    → WASM decides what to do → calls emit_signal() → UI receives signal
```

## Key Findings from Exploration

### 1. Holochain's Approach
- `recv_remote_signal` is a **regular zome function** exposed via `#[hdk_extern]`
- When a remote signal arrives, Holochain calls it like an RPC: `cell.call_zome("recv_remote_signal", payload)`
- The WASM decides whether to forward to UI via `emit_signal()`
- Typical implementation:
  ```rust
  #[hdk_extern]
  fn recv_remote_signal(signal: ExternIO) -> ExternResult<()> {
      emit_signal(&signal)?;  // Forward to UI
      Ok(())
  }
  ```

### 2. Current Fishy Implementation Issues
- `handleRemoteSignal()` in background/index.ts (line 1542) sends signals directly to all tabs
- Signal format is `type: "remote_signal"` with `dna_hash/from_agent` instead of proper AppSignal
- WASM never sees incoming remote signals
- ZomeClient filtering fails because signal type is wrong

### 3. Infrastructure Already Exists
- `executeZomeCallViaOffscreen()` can invoke any zome function
- `post_commit` callback pattern shows how to call WASM callbacks
- `emit_signal` host function already formats signals correctly for UI

## Implementation Plan

### Phase 1: Modify Background Remote Signal Handler

**File**: `packages/extension/src/background/index.ts`

**Change `handleRemoteSignal()` to invoke WASM instead of forwarding to UI:**

```typescript
async function handleRemoteSignal(signalData: {
  dna_hash: string;
  from_agent: string;
  zome_name: string;
  signal: number[];
}): Promise<void> {
  console.log(`[Background] Remote signal from ${signalData.from_agent} (${signalData.zome_name})`);

  try {
    // Find the context for this DNA
    const happContextManager = getHappContextManager();
    const contexts = await happContextManager.listContexts();

    // Find context with matching DNA hash
    const context = contexts.find(ctx =>
      encodeHashToBase64(ctx.dnaHash) === signalData.dna_hash
    );

    if (!context) {
      console.warn(`[Background] No context found for DNA: ${signalData.dna_hash}`);
      return;
    }

    // Build zome call request to invoke recv_remote_signal
    const zomeCallRequest: ZomeCallRequest = {
      cellId: [
        decodeHashFromBase64(signalData.dna_hash),
        context.agentPubKey  // Local agent receives the signal
      ],
      zome: signalData.zome_name,
      fn: "recv_remote_signal",
      payload: signalData.signal,  // Already msgpack-encoded
      provenance: decodeHashFromBase64(signalData.from_agent),  // Sender
    };

    // Execute the zome call (fire-and-forget)
    // This will invoke recv_remote_signal in the WASM
    // If WASM calls emit_signal(), the signal will be delivered to UI
    const tabId = /* find tab for this context */;
    await executeZomeCallViaOffscreen(context.id, zomeCallRequest, tabId);

    console.log(`[Background] recv_remote_signal invoked for ${signalData.zome_name}`);
  } catch (error) {
    console.error("[Background] Error invoking recv_remote_signal:", error);
  }
}
```

### Phase 2: Handle recv_remote_signal Payload Format

**File**: `packages/core/src/ribosome/index.ts`

The `recv_remote_signal` callback receives an `ExternIO` (msgpack-encoded bytes), not a complex struct. The WASM expects:
- Input: `ExternIO` (the signal payload, already msgpack-encoded)
- Output: `ExternResult<()>`

The current `callZome` already handles this - no changes needed to core ribosome.

### Phase 3: Fix TypeScript Type for recv_remote_signal Invocation

**File**: `packages/extension/src/background/index.ts`

Ensure the payload is passed correctly:
- `signalData.signal` is already `number[]` (msgpack bytes)
- Convert to `Uint8Array` before passing to executeZomeCallViaOffscreen
- The ribosome will wrap this as ExternIO for the WASM

### Phase 5: Ensure emit_signal Delivers to Correct Tab

**File**: `packages/extension/src/background/index.ts`

When `recv_remote_signal` is invoked, we need to know which tab to deliver any resulting `emit_signal` to. The tab ID must be tracked in the zome call context.

Currently, `executeZomeCallViaOffscreen` takes a `tabId` parameter, and signals emitted during that call are delivered to that tab. This should work as-is.

### Phase 6: Fix ZomeCallResult Type

**File**: `packages/core/src/ribosome/index.ts`

Add `remoteSignals` to the interface (or revert the earlier change if not needed for outgoing signals):

```typescript
export interface ZomeCallResult {
  result: unknown;
  signals: EmittedSignal[];
  pendingRecords?: HolochainRecord[];
  remoteSignals?: QueuedRemoteSignal[];  // Add this if we need to track outgoing

```

### Phase 7: Track Tabs for Signal Delivery

**Challenge**: When invoking `recv_remote_signal`, we need to know which tab(s) to deliver any resulting `emit_signal` to.

**Solution**: Query tabs by origin for the context's domain:
```typescript
// Find tabs that match this context's origin
const tabs = await chrome.tabs.query({ url: `${context.origin}/*` });
if (tabs.length === 0) {
  console.warn("[Background] No tabs found for context origin:", context.origin);
  return;
}

// Use the first tab (or deliver to all matching tabs by calling multiple times)
const tabId = tabs[0].id;
```

## Edge Cases to Handle

1. **WASM doesn't export recv_remote_signal**: Log warning, drop signal (fire-and-forget)
2. **No context for DNA**: Log warning, drop signal
3. **Multiple contexts for same DNA**: Use the most recently used context
4. **Tab no longer exists**: Signals emitted won't be delivered (existing behavior)

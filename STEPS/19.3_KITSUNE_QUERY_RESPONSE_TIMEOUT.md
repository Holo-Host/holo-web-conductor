# Step 19.3: Kitsune2 Query-Response Timeout (hc-membrane)

## Status: RESOLVED — Fixed in M4 milestone (kitsune-dht-ops branch, merged to agent-activity)

## Resolution Summary

Investigation on 2026-02-17 confirmed the query-response timeout issue described below was **resolved** as part of the M4 direct wire protocol work (kitsune-dht-ops branch, merged into agent-activity via commit `74c71f9`).

### Evidence

1. **M4_STATUS.md** (2026-02-05) shows successful round-trips:
   ```
   send_notify completed successfully, waiting for response... msg_id=223
   >>> Received GetLinksRes response msg_id=223
   Got GetLinksRes response msg_id=223 creates_count=1 deletes_count=0
   ```

2. **End-to-end plumbing confirmed**:
   - `DhtQuery` sends requests via `space.send_notify()`, registers oneshot channels in `PendingDhtResponses` keyed by `msg_id`
   - `ProxySpaceHandler::recv_notify()` decodes incoming responses and routes via `pending.route_response()`
   - Conductor (holochain) `recv_notify` processes queries and sends responses back via `space.send_notify(from_peer, ...)`
   - Both sides share the same `WireMessage` type from `holochain_p2p` — no encoding mismatch

3. **Wire encoding verified**: hc-membrane imports `holochain_p2p::WireMessage` directly (not a reimplementation), so encode/decode is identical by construction.

4. **No agent/gateway distinction**: Conductor does not distinguish between agent peers and non-agent peers in `recv_notify` — any peer with a URL and covering arc is a valid query target.

## Original Report (for reference)

### Symptom (now resolved)

Gateway sent `GetLinksReq` and `GetReq` (via `send_notify`) to conductors that had full arcs and were responsive (they accepted published data). However, conductors never sent a response within the 30-second timeout. Every DHT query returned empty after 30 seconds of idle waiting.

### Root Cause (now resolved)

The issue was in the M3-era implementation which lacked the complete request-response plumbing. The M4 milestone added:
- `PendingDhtResponses` shared map for correlating `msg_id` to oneshot channels
- `ProxySpaceHandler::handle_wire_message()` routing for `GetRes`/`GetLinksRes`/`CountLinksRes`/`ErrorRes`
- `PreflightCache` to include registered agent infos so conductors authorize gateway connections

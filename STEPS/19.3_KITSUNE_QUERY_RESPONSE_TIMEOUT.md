# Step 19.3: Kitsune2 Query-Response Timeout (hc-membrane)

## Status: Open — CRITICAL — Blocks all real multi-node operation (promoted to Step 24)

**Why critical**: Fishy nodes are zero-arc (hold no DHT data). The current ziptest e2e only passes because both agents share a single gateway whose REST endpoints serve locally-managed conductor data. In any real deployment with multiple gateways or external conductors, all cross-agent data retrieval depends on the kitsune2 query path documented here. Without this fix, fishy cannot function as a production system.

## Repo: hc-membrane (kitsune-dht-ops branch)

## Symptom

Gateway sends `GetLinksReq` and `GetReq` (via `send_notify`) to conductors that have full arcs (`Arc(0, 4294967295)`) and are responsive (they accepted published data). However, conductors never send a response within the 30-second timeout. Every DHT query returns empty after 30 seconds of idle waiting.

### Gateway Log Evidence

```
# Queries find responsive remote agents:
dht_get_links: finding peers for dna=uhC0kXKZ...
Found 2 responsive remote agents (of 2 total) for uhC0kXKZ...

# But every query takes exactly 30s with no response:
dht_get_links: time.busy=109µs time.idle=30.0s

# Meanwhile, publishing TO the same conductors works fine:
publish: Sent 12 ops to 2 peers
```

### Impact

- Every callZome that triggers a DHT query blocks the sync XHR worker for 30 seconds
- This serializes across all pending queries, causing cascade of timeouts
- Makes the extension non-functional for any cross-agent data retrieval

## Root Cause Analysis

### What Works

Publishing works because it uses `send_notify` in fire-and-forget mode — gateway sends `PublishReq` with DhtOps, conductors ingest them, no response required.

### What Doesn't Work

Querying requires a request-response pattern:
1. Gateway sends `GetLinksReq { msg_id, to_agent, link_key, options }` via `send_notify`
2. Conductor should process the query and send back `GetLinksRes { msg_id, response }` via `send_notify`
3. Gateway's `recv_notify` handler routes the response to the waiting future via `msg_id`

The breakdown is somewhere in this chain. Possible failure points:

1. **Conductor doesn't recognize the wire message format** — hc-membrane may be encoding `GetLinksReq` differently than what holochain expects
2. **Conductor processes but doesn't send response** — the kitsune2 `send_notify` handler in holochain may not be wired to send `GetLinksRes` back to non-agent peers (gateway is not a DHT agent)
3. **Response sent but not routed** — `recv_notify` handler may not match the response message variant

## Investigation Plan

### 1. Check conductor-side processing

Enable TRACE-level logging on the conductor to see if `GetLinksReq` messages arrive and are processed:

```bash
RUST_LOG=holochain_p2p=trace,kitsune2=trace ./holochain ...
```

Look for:
- Wire message deserialization of the incoming notify
- Query execution against the DHT store
- Whether a response message is constructed and sent back

### 2. Verify wire message encoding

Compare the msgpack encoding of `GetLinksReq` produced by hc-membrane vs what holochain produces in its own peer-to-peer queries. The wire protocol is defined in `holochain/crates/holochain_p2p/src/types/wire.rs`.

Key: Make sure hc-membrane's `WireMessage` enum encoding matches holochain's. Serde's internally-tagged enum format must be identical.

### 3. Check recv_notify routing in hc-membrane

In `src/gateway_kitsune.rs`, verify the `recv_notify` handler routes `GetRes` and `GetLinksRes` correctly to the pending futures map keyed by `msg_id`.

### 4. Test with a simple round-trip

Create a minimal test that:
1. Publishes a known entry to conductor
2. Sends a `GetReq` for that entry's hash
3. Checks if response arrives within timeout

## Key Files

| File | Repo | Purpose |
|------|------|---------|
| `src/dht_query.rs` | hc-membrane | Sends GetReq/GetLinksReq, awaits response |
| `src/gateway_kitsune.rs` | hc-membrane | `recv_notify` handler routes responses |
| `crates/holochain_p2p/src/types/wire.rs` | holochain | Wire protocol message definitions |
| `crates/holochain_p2p/src/lib.rs` | holochain | Conductor-side message handling |

## Success Criteria

- `GetLinksReq` returns `GetLinksRes` within <5 seconds
- `GetReq` returns `GetRes` within <5 seconds
- Gateway log shows `time.idle` < 5s (not 30s) for DHT queries
